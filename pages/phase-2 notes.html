Java Interview Notes - Phase 2

Exception Handling
Exception Hierarchy
Every exception in Java is an object. The root of the hierarchy is Throwable. Only objects that are instances of Throwable or its subclasses can be thrown or caught. Catch blocks must be arranged most-specific-first.
Checked vs Unchecked Exceptions
Checked exceptions extend Exception but NOT RuntimeException. The compiler forces you to handle them with try-catch or declare them with throws. Unchecked exceptions extend RuntimeException and the compiler does NOT enforce handling. They typically signal programming errors.
Error vs Exception
Both extend Throwable. Error represents serious problems that an application should NOT try to catch, like OutOfMemoryError and StackOverflowError. Exception represents application-level problems that programs should handle. Error is unchecked even though it's not a RuntimeException.
try-catch-finally
try is mandatory and must be followed by at least one catch OR finally or both. Multiple catch blocks must be ordered specific to general. At most one catch block runs per exception. finally runs regardless of whether an exception was thrown or caught, even after a return statement.
throw vs throws
throw is a statement used inside a method body to explicitly raise an exception. throws is a declaration in the method signature indicating which checked exceptions the method may propagate. Subclass overrides can declare fewer or narrower checked exceptions, never broader ones.
Custom Exception
Creating your own exception types makes error handling more expressive. Extend Exception for checked or RuntimeException for unchecked. Provide standard constructors and include meaningful context in the message. Name should end in Exception.
try-with-resources
Automatically closes resources like files, streams, and database connections even if exceptions occur. The resource must implement AutoCloseable. Resources are closed in reverse order of declaration. If close throws while the try block already threw, the close-time exception is added as suppressed.
finally Block Behavior
finally is designed for cleanup. It always executes after normal completion, after caught or uncaught exceptions, and even after a return. It does NOT execute when System.exit is called, the JVM crashes, or a deadlock occurs. Never put return or throw inside finally as it masks bugs.
JVM Internals
JVM Architecture
The JVM is the runtime engine that executes Java bytecode. Its three main subsystems are: Class Loader Subsystem for loading classes, Runtime Data Areas for memory management including Method Area, Heap and Stack, and the Execution Engine with Interpreter, JIT Compiler and Garbage Collector.
ClassLoader
Class loaders load .class files into the JVM on demand using lazy loading. Java uses a hierarchical parent-delegation model: Bootstrap loads core classes, Extension/Platform loads non-core platform modules, and Application loads user classes from the classpath. A class loader always delegates to its parent first.
Method Area / Metaspace
The Method Area is a JVM memory region shared across all threads that stores class-level metadata, method bytecode, runtime constant pool, and static variables. Pre-Java 8 it was called PermGen with fixed size. Java 8+ it's called Metaspace and lives in native memory, auto-growing by default.
JIT Compiler
The Just-In-Time compiler translates bytecode to native machine code at runtime, dramatically boosting performance after warmup. When a method crosses an invocation threshold of about 10,000 calls, JIT compiles it to native code. Optimizations include method inlining, escape analysis, dead code elimination, and loop unrolling.
Execution Engine
Executes the bytecode loaded by the class loader. Has three main components: the Interpreter which reads bytecode one instruction at a time, the JIT Compiler which compiles hot methods to native code, and the Garbage Collector which reclaims memory. Also includes the Native Method Interface for calling C and C++ libraries.
Garbage Collection
GC frees memory held by objects no longer reachable from any GC root. Based on the generational hypothesis that most objects die young. New objects go to Eden, surviving objects move through Survivor spaces, and long-lived objects get promoted to Old Generation. Default collector since Java 9 is G1.
Object Lifecycle
An object goes through allocation in Eden with new keyword, initialization via the constructor chain from Object down, usage by the application, unreachability when all references are gone, optional finalization if finalize is overridden, and finally reclamation by the garbage collector.
Java 8+ Features
Lambda Expressions
A lambda is an anonymous function, a concise way to represent a single method interface called a functional interface. Captured local variables must be effectively final. The this keyword refers to the enclosing class. Lambdas use invokedynamic bytecode for efficient implementation, not inner classes.
Functional Interfaces
An interface with exactly one abstract method, also known as Single Abstract Method or SAM. It's the target type for lambdas and method references. Key built-in ones include Function for transformation, Predicate for testing conditions, Consumer for side effects, and Supplier for producing values.
Streams API
A stream is a sequence of elements supporting aggregate operations in a declarative, functional style. Streams don't store data, they describe a computation pipeline. They use lazy evaluation where intermediate operations do nothing until a terminal operation triggers the pipeline. Streams are single-use.
Stream vs Collection
Collections are for data storage with eager evaluation and can be traversed multiple times. Streams are for data computation with lazy evaluation and can be traversed only once. Collections use external iteration with for-each loops while streams use internal iteration where the API controls traversal.
Intermediate vs Terminal Operations
Intermediate operations return a new Stream and are lazy. They can be stateless like filter and map, or stateful like sorted and distinct. Terminal operations trigger pipeline execution and produce a non-stream result. Short-circuiting operations like findFirst can complete without processing every element.
Optional
A container that may or may not hold a non-null value. Designed to reduce NullPointerException and make nullability explicit in the API. Use as return type where absence is valid. Don't use as fields, method parameters, or in collections. orElseGet is preferred over orElse when the default is expensive.
Method References
Shorthand syntax for lambdas that simply invoke a single method using the double colon operator. Four forms: static method reference like Integer parseInt, bound instance method reference, unbound instance method reference like String length, and constructor reference like ArrayList new.
Default & Static Methods in Interface
Default methods have a body defined using the default keyword, enabling backward-compatible API evolution. Static methods belong to the interface itself and are not inherited. When a class inherits the same default method from multiple interfaces, the class must override and can call either with InterfaceName.super.
Serialization
Serialization & Deserialization
Serialization converts an object's state into a byte stream for persistence, network transfer, or caching. Deserialization reverses the process. The JVM walks the object graph, serializing each reachable non-transient field. Constructor does NOT run during deserialization. Never deserialize untrusted data.
Serializable Interface
A marker interface with no methods signaling to the JVM that instances of this class may be serialized. All non-transient, non-static fields must themselves be serializable. If a superclass is not serializable but a subclass is, the superclass must have a no-arg constructor.
transient Keyword
Marks a field to be excluded from default serialization. Upon deserialization, the field will have its default value: null for objects, 0 for numbers, false for booleans. Common uses include sensitive data like passwords, derived or cached fields, and non-serializable resources like database connections.
serialVersionUID
A version identifier for serialized classes. Used to verify that serialized data is compatible with the class definition during deserialization. Without it, any class change causes InvalidClassException. Must be private static final long. Always declare it explicitly.
Externalizable
A sub-interface of Serializable giving the developer full control over the serialization format. Requires implementing writeExternal and readExternal methods. Unlike Serializable, the public no-arg constructor IS called during deserialization. Faster than Serializable because it avoids reflection.
Enum
Enum Basics
An enum is a special class that represents a fixed, finite set of constants. Each enum value is an instance of the enum type, implicitly public, static, and final. Enums extend java.lang.Enum and cannot extend any other class but can implement interfaces. You cannot instantiate an enum with new.
Enum vs Constants
Enums provide type safety where the compiler rejects non-enum values. They are self-documenting, support iteration via values method, and can carry data and behavior. Plain int constants are not type-safe, can clash, print as numbers, and cannot iterate or add behavior.
Enum with Methods and Fields
Enums can have fields, constructors, and methods, making them essentially full classes with a fixed set of instances. They support constant-specific method bodies for overriding per constant, and can implement interfaces. The Enum Singleton Pattern is the best singleton approach being serialization-safe, reflection-safe, and thread-safe.
Reflection API
What is Reflection
Reflection is the ability of a Java program to inspect and modify its own structure at runtime without knowing class names at compile time. Core classes include Class, Method, Field, and Constructor. You can dynamically instantiate objects, invoke methods, and access fields using setAccessible.
Use Cases of Reflection
Used in frameworks like Spring for dependency injection, Hibernate for ORM mapping, Jackson for JSON serialization, and JUnit for finding test methods. Also used for dynamic proxies in AOP, IDE features like auto-complete, plugin architectures, and testing tools like Mockito.
Performance Impact of Reflection
Reflection is 10 to 100 times slower than direct method calls due to access checks, method lookup by name, no JIT inlining, boxing of primitives, and varargs array allocation. Mitigate by caching Method and Field objects, calling setAccessible once, and using MethodHandle for new code.
Security Concerns with Reflection
Reflection can bypass standard access controls, access private fields, modify final fields, and break singletons. Modern safeguards include the Java 9+ Module System with strong encapsulation where reflection cannot access non-exported package internals even with setAccessible. Avoid reflection unless absolutely necessary.
Annotations
Built-in Annotations
Override provides compile-time checking of method overrides. Deprecated marks API as outdated with optional since and forRemoval elements. SuppressWarnings tells the compiler to ignore specific warnings. FunctionalInterface enforces the single abstract method rule. SafeVarargs suppresses unchecked warnings for generic varargs.
Custom Annotations
Type declarations you attach as metadata to program elements like classes, methods, and fields. Allowed element types are primitives, String, Class, enums, other annotations, and arrays of these. Meta-annotations include Retention for how long it's retained, Target for what elements can be annotated, Inherited, and Repeatable.
Runtime vs Compile-time Annotations
SOURCE retention is discarded after compilation and used by annotation processors like Lombok. CLASS retention is in the .class file but not loaded at runtime. RUNTIME retention is loaded into the JVM and queryable via reflection, driving most framework magic in Spring, JUnit, JPA, and Jackson.
IO & NIO
File Handling
Java has two file APIs. Legacy java.io.File has limited error handling and returns booleans. Modern java.nio.file introduced in Java 7 uses Path and Files classes with full exception-based error reporting, symbolic link awareness, and file attributes support. Use java.nio.file for new code.
InputStream / OutputStream
Byte streams are the base abstraction for binary data. Key methods are read and write. FileInputStream and FileOutputStream handle files. BufferedInputStream adds buffering. DataInputStream handles typed data. These streams illustrate the classic decorator pattern where you wrap a basic stream with decorators to add functionality.
Reader / Writer
Character streams designed for text, handling encoding from bytes to characters automatically. InputStreamReader wraps an InputStream with a charset to produce a Reader. OutputStreamWriter does the reverse. Always specify charset explicitly to avoid portability bugs. Prefer Files.newBufferedReader with UTF-8.
Buffered Streams
Wrap another stream to add an internal buffer, typically 8 kilobytes. Reading and writing in chunks dramatically reduces system calls. BufferedReader adds readLine for line-at-a-time reading. BufferedWriter adds newLine for platform-appropriate line separators. Modern alternative is Files.lines for streaming file lines.
NIO Basics
New IO is buffer-oriented and supports non-blocking operations. Buffers are blocks of memory with position, limit, and capacity. Channels are bidirectional data conduits for files and sockets. Selectors allow one thread to manage many channels. Used by Netty and Tomcat for high-concurrency servers. Memory-mapped files map files directly into memory.
Generics
Generic Classes and Methods
Generics parameterize types over other types, giving compile-time type safety and eliminating casts. A generic class uses type parameters like T. The diamond operator in Java 7+ infers type arguments. Naming conventions include T for Type, E for Element, K and V for Key and Value, and R for Return type.
Type Erasure
Java generics are a compile-time feature. At runtime, all type parameters are erased and replaced with their upper bound or Object if unbounded. This exists for backward compatibility. Consequences include no new T, no generic arrays, no instanceof with parameterized types, and no overloads differing only by generic type.
Bounded Types
Restrict type parameters to specific types or their subtypes using extends. Multiple bounds use type intersection with ampersand, allowing at most one class which must come first plus any number of interfaces. Bounded types let you call specific methods on T. Lower bounds on type parameters do not exist, only on wildcards.
Wildcards
Wildcards express flexibility in generic types around variance. Upper bounded ? extends T means T or any subtype and is a Producer. Lower bounded ? super T means T or any supertype and is a Consumer. The PECS mnemonic says Producer Extends, Consumer Super. Generics are invariant by default.
Immutable Objects
What Makes a Class Immutable
An immutable object's state cannot change after construction. Rules: no setters or state-modifying methods, make the class final, make all fields final and private, and make defensive copies of mutable internal state in both constructor and getters. Do validation after the defensive copy to prevent time-of-check to time-of-use bugs.
Benefits of Immutability
Thread-safety for free since no synchronization is needed. Safe to share and cache since state never changes. Safe as HashMap keys because hashCode is stable. Failure atomicity since the original object remains unchanged on failure. Easier to reason about with no who-modified-this-when debugging. Famous immutable classes include String, all boxed primitives, and java.time types.
Design Principles
SOLID Principles
Single Responsibility: a class should have one reason to change. Open/Closed: open for extension, closed for modification. Liskov Substitution: subtypes must be substitutable for base types. Interface Segregation: clients should not depend on interfaces they don't use. Dependency Inversion: depend on abstractions, not concretions.
DRY, KISS
DRY means Don't Repeat Yourself. Every piece of knowledge should have a single, unambiguous representation. But premature DRY can be worse than duplication. KISS means Keep It Simple, avoid unnecessary abstractions and over-engineering. Related is YAGNI, You Aren't Gonna Need It: don't add functionality until you actually need it.
High Cohesion vs Low Coupling
High cohesion means a module does one thing well. Low coupling means modules interact through minimal, well-defined interfaces. Goal is high cohesion plus low coupling for maintainability, testability, reusability, and readability. Achieved through dependency injection, single responsibility, interfaces, and event-driven patterns.
Object Class Methods
toString()
Returns a String representation of the object. Default is ClassName@hexHashCode. Override to include the class's most important fields. Automatically called by println, string concatenation, String.format, debuggers, and loggers. Makes debugging enormously easier.
equals() & hashCode()
equals compares logical equality and must be reflexive, symmetric, transitive, and consistent. hashCode generates an integer for bucket placement. The mandatory contract is: if x equals y then their hashCodes must be equal. Violations break HashMap and HashSet. If you override equals, you MUST override hashCode.
clone()
Creates a copy of the object. Requires the class to implement Cloneable marker interface, else throws CloneNotSupportedException. super.clone does a shallow copy where object references point to the same sub-objects. For deep copy, manually clone referenced objects. Prefer copy constructors or copy factory methods over clone.
finalize()
Called by the garbage collector before reclaiming an object's memory. Deprecated since Java 9 because timing is unpredictable, performance cost is high, and it can resurrect objects. Modern replacements are try-with-resources, AutoCloseable interface, and java.lang.ref.Cleaner introduced in Java 9.
getClass()
Returns the runtime Class object of this object. Entry point to reflection. It is a final method and cannot be overridden. Difference from .class literal: String.class is compile-time while obj.getClass() is runtime and returns the actual class, which is important for polymorphism.
wait(), notify(), notifyAll()
Low-level thread coordination methods that must be called while holding the object's monitor inside synchronized blocks. wait releases the monitor and enters the wait set. notify wakes one arbitrary waiting thread. notifyAll wakes all waiting threads. Always use wait in a while loop due to spurious wakeups. Modern alternatives include Lock with Condition and BlockingQueue.
Thread Advanced Topics
Executor Framework
Decouples task submission from thread management. Core interfaces include Executor with execute method, ExecutorService with submit and shutdown, and ScheduledExecutorService for delayed or periodic tasks. Standard factories provide fixed, cached, single-thread, and work-stealing pools. Always shutdown to prevent JVM from hanging.
Thread Pool
Reuses a limited set of worker threads for many tasks. ThreadPoolExecutor has configurable core pool size, maximum pool size, keep-alive time, work queue, and rejection policy. Rejection policies include AbortPolicy which throws, CallerRunsPolicy for backpressure, and DiscardPolicy. Size CPU-bound tasks at cores plus one, IO-bound tasks much higher.
Callable & Future
Callable returns a value and can throw checked exceptions, unlike Runnable. Future is a handle to an asynchronous computation with get that blocks until done, cancel, isDone, and isCancelled. CompletableFuture in Java 8 adds composition with thenApply, thenCompose, thenCombine, and exceptionally for error handling.
ForkJoinPool
A specialized pool for divide-and-conquer parallelism. Each worker thread has its own deque of tasks and steals from other workers when idle. Core classes are RecursiveTask for results and RecursiveAction for void. The common pool is shared by parallelStream and CompletableFuture. One long-running task can starve other operations JVM-wide.
BlockingQueue
A thread-safe queue that blocks callers when operations can't proceed. put blocks if full, take blocks if empty. Implementations include ArrayBlockingQueue for bounded FIFO, LinkedBlockingQueue for higher throughput, PriorityBlockingQueue for priority ordering, DelayQueue for delayed elements, and SynchronousQueue for direct handoff with zero capacity.
Producer-Consumer Problem
One or more producers generate items while one or more consumers process them. The idiomatic solution uses BlockingQueue which handles bounded buffer and coordination automatically, eliminating wait and notify boilerplate. Alternative solutions include wait/notify for educational purposes and Lock with separate Condition objects for more control.
Date & Time API
LocalDate, LocalDateTime
Java 8's java.time package provides clean, immutable, type-safe date and time API. LocalDate is date without time or zone. LocalTime is time only. LocalDateTime is date plus time. ZonedDateTime includes timezone. Instant is nanoseconds since epoch in UTC. All types are immutable and thread-safe with fluent API.
Old Date vs New API
Legacy java.util.Date and Calendar are mutable and not thread-safe. Month indexing is 0-based with January as 0. SimpleDateFormat is NOT thread-safe. New java.time is immutable, uses 1-based months, and DateTimeFormatter is thread-safe. For new code use java.time and wrap legacy APIs at boundaries.
Time Zones
ZoneId represents a region like Asia/Kolkata and handles daylight saving time. ZoneOffset is a fixed offset from UTC like +05:30. Store timestamps as UTC Instant and convert to user's zone only for display. Be explicit about zone in logs. DST can make times non-existent or ambiguous.
Class Design Concepts
Immutable vs Mutable Classes
Mutable state can change after construction, suitable for entities with identity like JPA User. Immutable state is fixed at construction, suitable for value objects like Money and Address. Make classes immutable unless there's a concrete reason not to. Use Java records for quick immutable data classes. Use builder pattern for constructing complex immutable objects.
Singleton Pattern
A class that guarantees exactly one instance per JVM with global access. Best approach is Enum Singleton which is serialization-safe, reflection-safe, and thread-safe. Bill Pugh Holder uses lazy class loading. Double-Checked Locking requires volatile. Eager Initialization is simple but not lazy. Enum avoids all pitfalls automatically.
Factory Pattern
Encapsulates object creation, returning objects of some type without exposing instantiation logic. Variants include Simple Factory with a static method, Factory Method where subclasses decide, and Abstract Factory for families of related objects. Benefits include hiding concrete types, enabling caching, and simplifying testing with mock factories.
Misc Frequently Asked
final, finally, finalize
Three unrelated things. final is a keyword: final class cannot be subclassed, final method cannot be overridden, final variable cannot be reassigned but for object references the object's state may still mutate. finally is a block that always executes after try-catch for cleanup. finalize is a deprecated method on Object called by GC.
transient vs volatile
transient is a serialization modifier meaning don't persist this field. Upon deserialization the field has its default value. volatile is a concurrency modifier guaranteeing visibility of writes across threads. volatile does NOT guarantee atomicity of compound operations like count++. Use AtomicInteger for atomic operations.
instanceof
Operator that tests whether an object is an instance of a type. Returns false for null without throwing NullPointerException. Parameterized types not allowed like List<String> but unbounded wildcard is OK. Java 16 adds pattern matching to eliminate casts. Java 21 adds pattern matching in switch. Overuse is a code smell, prefer polymorphism.
Marker Interfaces
An interface with no methods or constants used purely to signal that a class has a property. Built-in examples include Serializable, Cloneable, and RandomAccess. Modern alternative is annotations which can target non-class elements and carry parameters. Prefer marker interfaces when you need a type for polymorphism or compile-time checks.
